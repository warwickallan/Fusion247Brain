\# F247.skill.process-youtube-transcript

Version: v0.1. Created: 2026-07-10. Owner: Warwick. Layer: Skill/SOP (myPKA SOP equivalent \- discrete, named, callable).  
Default owner agent: CategorisAIr. Invocable by: any agent, Claude, or GPT/Fusion. This is NOT an agent and NOT CategorisAIr. It makes no filing, promotion, template-selection or backlink decisions \- it prepares; CategorisAIr decides.  
Location: 00\_System/skills. Registered in F247 Drive Object Registry.

\#\# Purpose  
Convert one captured YouTube transcript (Youtubair markdown source: YAML frontmatter \+ timestamped transcript) into a Brain-ready knowledge packet: one Obsidian-grade knowledge note plus justified entity candidates plus register candidates, chunked, provenance-anchored, marked pending\_categorisair. Maximum knowledge capture, minimum token burn.

\#\# SSOT references (rules live there, not here \- do not restate them in outputs)  
\- Content rules (no knowledge loss / not a summary / not a transcript; what to drop; note naming): F247.template.youtube-transcript-knowledge-note  
\- Template selection and classification: F247.template.source-classification-registry (CategorisAIr's job, done BEFORE this skill runs)  
\- Backlink justification test (no \[\[Coffee\]\] stubs; enrich \[\[NetworkChuck\]\]): rule as recorded in the template \+ classification registry  
\- Register routing (actions \-\> task ledger; open questions \-\> open-question register; candidate decisions \-\> decision log; verification needs \-\> VerifiAIr): F247.agent.categorisair routing correction 2026-07-02

\#\# Preconditions  
1\. Source classified as public YouTube/video transcript (not a work/client meeting \- those use the Bellrock project extraction skill).  
2\. RAW transcript preserved immutable (02\_Sources/youtube/transcripts/ or Youtubair intake path). This skill reads RAW once and never edits it.  
3\. Source metadata present (video\_id, channel, url, captured\_at). Missing metadata is flagged, never invented.

\#\# Procedure (single-read chunking)  
1\. ONE sequential read of the transcript. Segment into knowledge chunks at natural topic boundaries \- a chunk is one teachable idea, workflow, tool walkthrough, claim cluster or argument. While reading, capture per chunk, inline, once:  
   \- theme label (2-5 words)  
   \- knowledge points in dense prose (facts, steps, commands, numbers, versions, trade-offs, caveats \- no filler)  
   \- entities touched (people, tools, products, orgs, concepts)  
   \- claims needing verification (popularity stats, benchmarks, garbled names)  
   \- anchor \[mm:ss-mm:ss\]  
   The chunk map is the working artifact from here on. The transcript is not re-read.  
2\. Assemble the knowledge note per the template:  
   \- YAML frontmatter per the contract below (flat keys \- built for SQLite columns and Obsidian, not prose)  
   \- Orientation: max 8 lines answering why this matters to the Brain / Warwick  
   \- Knowledge reconstruction organised by THEME using the chunk map. Reorder from video order where retrieval improves; preserve sequence where sequence is the knowledge (installs, workflows)  
   \- Each theme section ends with its anchor(s)  
   \- Verification-needed list (from chunk claims)  
   \- Transcript-noise flags: auto-caption garble gets a low-confidence marker, never silent correction of names/products  
3\. Entity candidates \- from the chunk entity lists only (no re-read). Apply the backlink justification test. Output two lists:  
   \- ENRICH: existing WIKI notes this source adds real context to (1 line of what to add, \+ anchor)  
   \- NEW: candidate notes where the source carries enough knowledge to seed a non-stub note (1-line justification each)  
   Passing mentions are attached under their parent entity, not proposed as notes.  
4\. Register candidates: actions, open questions, candidate decisions, verification needs \- as a list for CategorisAIr to route. This skill does not write to any register.  
5\. Coverage check \- run over the CHUNK MAP, not the transcript: every named tool/product/person accounted for; numbers, versions and commands captured; caveats and trade-offs present; sequences intact where they matter; sponsor/intro/outro dropped unless substantive.

\#\# Token-efficiency rules (hard)  
1\. One full read only. Coverage runs on the chunk map.  
2\. Never reproduce transcript blocks. Quotes under 15 words, and only where the wording IS the knowledge.  
3\. Anchors at chunk level, never per sentence.  
4\. No entity stubs that fail the justification test \- empty stubs cost tokens forever.  
5\. Do not copy template/registry rules into outputs; wikilink them.  
6\. Frontmatter stays flat and fixed \- it feeds parsers, not readers.  
7\. Transcripts over \~90 minutes: process in sequential slices, carrying only the chunk map forward, never raw text.  
8\. Do not run a hostile-panel/QA pass by default; that is a separate skill invoked on request.

\#\# Frontmatter contract (SQLite \+ Obsidian ready; aligns with Youtubair capture YAML and SOP-002 column mapping)  
title, source\_type: youtube\_transcript, source\_url, video\_id, channel, captured\_at, processed\_date, processed\_by, template: F247.template.youtube-transcript-knowledge-note, raw\_source: \[\[...\]\], knowledge\_domain, maturity, confidence, review\_status: pending\_categorisair, tags: \[\]

\#\# Output packet (handed to CategorisAIr, review\_status pending\_categorisair)  
1\. Knowledge note (short unique content title per template naming doctrine)  
2\. Entity candidate lists (ENRICH / NEW with justifications)  
3\. Register candidate list  
4\. Verification \+ uncertainty list  
This skill updates no living knowledge, creates no WIKI backlinks, files nothing.

\#\# Failure modes  
\- Transcript unavailable/garbled beyond use: return failure record, no partial note pretending completeness.  
\- Metadata missing: flag as unknown-in-source.  
\- Ambiguous entity identity (which "Jeff"?): flag, never guess.
